# PROJECT_MAP.md — mergesavedcart

## [TECH_STACK]
- PrestaShop 9.x (module declares `ps_versions_compliancy` min `9.0.0`)
- PHP 8.1+ (host environment; no PHP version pin in this module beyond PrestaShop's own requirement)
- Symfony 6.4 LTS (PrestaShop 9's bundled version) — used here only to **decorate** an existing core service (`prestashop.adapter.customer.customer_configuration`), not to build a new admin form
- MySQL — no new tables; reuses core `cart`, `cart_product`, `orders`, and the core `log` table (via `PrestaShopLogger`)
- No frontend build step: Bootstrap 5.3 (PrestaShop 9's native front theme framework) + vanilla JS, no bundler

## [SYSTEM_FLOW]

### Rule 1 — BO lock on "Re-display cart at login"
The brief referred to this setting as `PS_REPLAY_CART`; the real configuration key in this codebase is **`PS_CART_FOLLOWING`** (label "Re-display cart at login" / FR "Réafficher le panier après identification"), set by `src/Adapter/Customer/CustomerConfiguration.php` via the Symfony "Customer Settings" form (`CustomerPreferencesController` → `CustomerPreferencesDataProvider` → `prestashop.adapter.customer.customer_configuration`).

**Why it must stay off:** `Context::updateCustomer()` (core, `classes/Context.php:304-310`) checks this flag on every login and, if enabled, silently replaces the customer's current cart with their last unordered cart *before* any hook fires — bypassing this module's opt-in restoration modal entirely and creating two competing "cart merge" mechanisms.

**Enforcement (two layers, both implemented — a third, broader layer was considered and deliberately dropped, see below):**
1. **Prevents the write**: `src/Configuration/CustomerConfigurationGuard.php` decorates `prestashop.adapter.customer.customer_configuration` (`config/admin/services.yml`, service id `mergesavedcart.customer_configuration_guard`), forcing `redisplay_cart_at_login` to `false` on every read and write — the same service-decoration pattern already used by the `optimizer` reference module for its language form (down to the file location: `config/admin/services.yml`, not a module-root `config/services.yml`). This is the layer that actually stops `PS_CART_FOLLOWING` from ever being persisted as `true`, with zero window where the wrong value could be live in the `configuration` table.
2. **Explains the block to the admin**: `hookActionCustomerPreferencesPageSave()` in `mergesavedcart.php`. Found by reading `src/Core/Form/Handler.php::save()` — the generic Symfony form-handler base class every "Configure" page uses — which dispatches a hook named `action{$hookName}Save` (here `$hookName = 'CustomerPreferencesPage'`, set in `config/services/bundle/form/form_handler.yml`) with `['errors' => &$errors, 'form_data' => &$data]`, **both passed by reference**. Important ordering fact: `formDataProvider->setData($data)` (the actual write, going through the decorator above) runs **before** this hook fires — so the hook cannot itself prevent persistence, only react to it. It checks `$params['form_data']['redisplay_cart_at_login']` (what the admin actually submitted) and, if truthy, pushes a translated message into `$params['errors'][]`, which the controller (`CustomerPreferencesController::processAction()`) surfaces via `addFlashErrors()` instead of the generic success message.
- **Dropped**: an earlier revision also had `hookActionAdminControllerSetMedia()` re-checking `Configuration::get('PS_CART_FOLLOWING')` on *every* BO page load as a catch-all against out-of-band writes (raw SQL, another module, CLI) that don't go through this Symfony form at all. Replaced by the two layers above per explicit instruction — accepted trade-off: a write to `PS_CART_FOLLOWING` that bypasses the Customer Preferences form entirely (e.g. a raw `Configuration::updateValue()` call from unrelated code) is no longer caught by this module.

### Rule 2/3 — Detection and display both happen at page render, id passed to the client (no cookie)
**Design history (four revisions; this is the current, final one):**
1. First version fired on `actionAuthentication` and stashed the full eligible-products payload in a cookie (mirroring `payzen_embedded`'s wallet-controller flag pattern).
2. Second version removed the cookie entirely and recomputed everything live on every page view, memoized only within the current PHP request.
3. Third version kept `actionAuthentication` firing once at login, but slimmed the cookie down to carry only the abandoned cart's **id** (a single int) — never the product list — with `hookDisplayModalContent()` reading that cookie on every subsequent page view.
4. **Current version, per explicit instruction: the cookie is gone entirely.** `hookActionAuthentication()` was removed (no longer registered in `install()`), and `MergeSavedCart::ABANDONED_CART_COOKIE_KEY` no longer exists. `hookDisplayModalContent()` now calls `AbandonedCartFinder::findAbandonedCartId()` **on every render** (same live-recompute approach as revision 2, but combined with rendering in the same call rather than two separate steps) and, if found, hands the id straight to the browser as a `data-id-abandoned-cart` attribute on the modal (`mergesavedcart_abandoned_cart_id` smarty var, set in `restore-cart-modal.tpl`). `restore-cart-modal.js` reads that attribute and includes it as `id_abandoned_cart` in every POST to `restore.php`, which trusts it only after re-verifying ownership server-side (`AbandonedCartFinder::deleteAbandonedCart()`'s `id_customer` check) — the same security boundary that protected the cookie-encrypted value before, just applied to a plain client-submitted param instead. This removes all the `Cookie::write()` plumbing revision 3 needed (login redirects and `ajax = true` controllers skipping `smartyOutputContent()`), since nothing is written to a cookie anywhere in this flow anymore.

**TLD scoping (`nw_multidomainmanager` integration, added after the four revisions above):** this shop runs `nw_multidomainmanager`, which makes one PrestaShop install serve several independent storefronts ("TLDs", one per domain) — product availability and pricing differ per TLD (`nw_tld_product`/`nw_tld_product_attribute`). A cart started on one domain must never be proposed for restoration on another. `AbandonedCartFinder` guards every TLD touch with `Module::isEnabled('nw_multidomainmanager')` (same convention `wwwnutriconnect` already uses), so the module works unmodified on any install where `nw_multidomainmanager` isn't present.
- `findAbandonedCartId()` joins the candidate cart against `nw_order` (the module's cart→TLD association table, `id_nw_tld`/`id_cart`) and restricts to the current request's TLD (`Context::getContext()->shop->nwTld->id`, via `NwTld`). If `nw_multidomainmanager` is enabled but the current request has no resolved TLD (unmapped host), nothing is proposed at all — a resolution failure is not treated as "no restriction."
- `isValidAbandonedCart()` (called from `findEligibleProducts()` at display time) re-checks the same TLD match via `isSameTld()`, on top of its existing owner/order/product-count checks — same defense-in-depth pattern as the rest of that method.
- **A cart with no `nw_order` row at all does not match, ever** — not even against another request that also has no resolved TLD. `nw_order` rows are only written by `nw_multidomainmanager`'s `hookActionCartSave()`, which itself no-ops when no TLD is active for that write (pre-module carts, back-office/webservice-created carts, or an unmapped host) — so a cart with no row means "unknown TLD," and `null === null` is never treated as a match here.

Flow:
1. `hookDisplayModalContent()` calls `AbandonedCartFinder::findAbandonedCartId($idCustomer, $idCurrentCart)` — the customer's most recent cart, other than the current one, with no linked order and at least one product row **and, when `nw_multidomainmanager` is active, on the same TLD as the current request** — directly, on every page view. If it holds an id, it calls `AbandonedCartFinder::findEligibleProducts($idCustomer, $idAbandonedCart, $idCurrentCart)`, which:
   - Re-validates the cart still qualifies (`isValidAbandonedCart()`: right owner, no order since, still has products, and — when `nw_multidomainmanager` is active — same TLD as the current request) — returns `null` otherwise.
   - Diffs its product lines against the *current* cart's live contents, **by `id_product`** — a product already present in the current cart in any variant is excluded entirely, per the brief's "n'existent pas du tout" wording. This covers both Scenario 1 (current cart has products) and Scenario 2 (empty current cart) with the same code path.
   - This diff is deliberately never cached: only the id survives between requests, so the comparison always reflects the current cart's live state.
3. If the diff comes back empty (cart no longer valid, or every product already duplicated), nothing renders. Otherwise the modal renders with `AbandonedCartFinder::presentProducts()` (name/image/combination/quantity, same `cart_default` image type as the core cart template). **No price is shown at all** — an earlier revision computed pre-formatted `unit_price`/`total_price` strings (via `Tools::getContextLocale($context)->formatPrice(...)`, since there is no plain static `Tools::displayPrice()` callable from Smarty, only the Smarty-plugin-specific `displayPriceSmarty()`), but the price line was dropped from the template per explicit instruction, and the now-dead pricing code (`Product::getPriceStatic()`, the `Tools` import) was removed from `presentProducts()` along with it.
   - **Combination label** (e.g. "30g • Chocolate"), shown under the product name when `id_product_attribute` is non-zero: `AbandonedCartFinder::getCombinationLabel()` queries `product_attribute_combination`/`attribute`/`attribute_lang`/`attribute_group` directly and joins the attribute value names with `' • '` itself. Deliberately **not** reusing `Cart::getProducts()`'s own `attributes_small` field (which is already computed on that row) — that field joins values with `PS_ATTRIBUTE_ANCHOR_SEPARATOR`, a shop-configurable SEO URL-anchor separator (`-` by default), not `•`, and isn't meant for this kind of display use.
   - **Combination-specific image**: `presentProducts()` tries `Product::getCombinationImageById($idProductAttribute, $idLang)` first (a variant, e.g. a different color, can have its own dedicated image via the `product_attribute_image` table) and only falls back to `Product::getCover()` (the product's generic cover) if the combination has none of its own or `id_product_attribute` is `0`.
   - **Quantity badge**: `<span class="badge badge-quantity sm">{$product.quantity}</span>` overlaid on the product image (wrapped in a plain Bootstrap `position-relative` div), matching the exact class names the theme itself uses for the same purpose in `themes/custom/templates/checkout/_partials/cart-summary-product-line.tpl`. Caveat: that theme's `.badge.badge-quantity { right: -3px; top: -6px; }` positioning rule (`src/scss/prestashop/components/cart/_cart-summary-product.scss`) is nested under the `.cart-summary-product` component class, which this module's markup doesn't use — the badge gets the base `.badge`/`.sm` look but not necessarily that exact corner offset outside of that specific theme component's context.
4. `hookActionFrontControllerSetMedia()` unconditionally enqueues `views/js/restore-cart-modal.js` on every front page (no cookie check, no DB query) — the modal itself may still end up not rendering if there's no abandoned cart or the cart turns out invalid at display time, in which case the script just no-ops (`getElementById('mergesavedcart-restore-modal')` returns `null`).
   - **No custom CSS / z-index override.** A `restore-cart-modal.css` bumping `#mergesavedcart-restore-modal` and `.modal-backdrop` z-index was added at one point (theory: this theme's highest overlay, `_cachebuster.scss`, sits at `z-index: 1560`, above Bootstrap's default `$zindex-modal: 1055`) and then removed per explicit instruction — the theme's own real modals (quickview, product-comments' review modals, both confirmed via live rendered HTML) use plain, unmodified Bootstrap `.modal`/`.modal-dialog`/`.modal-content` classes with no extra CSS at all and render correctly, so this module follows the same convention.
   - **The real cause of any stacking/visibility problem was the choice of hook, not CSS** — see below.

**Why `displayModalContent`, not `displayFooter` (this theme has a dedicated hook for exactly this):** `themes/custom/templates/layouts/layout-both-columns.tpl:107-114` has a `{block name='modal_container'}` containing `<div data-ps-target="modal-container">{hook h='displayModalContent'}</div>`, placed right before `</body>`, **after** the footer/javascript/bottom-elements blocks — i.e. outside the nested column/layout block structure entirely. `themes/custom/src/js/helpers/modal.ts`'s `getModalContentContainer()` (used by `quickview.ts`) targets that same `[data-ps-target="modal-container"]` selector. `displayFooter`, by contrast, renders deep inside the nested layout hierarchy (confirmed via a live stack trace: `layout-both-columns.tpl` → `layout-left-column.tpl` → `product-list.tpl` → `category.tpl`, many levels of `Smarty_Internal_Runtime_Inheritance` block nesting) — a materially different, and much deeper, DOM insertion point than where the theme's own modals (quickview, product-comments) actually render. Switched `hookDisplayFooter()` → `hookDisplayModalContent()`, and `install()`'s `registerHook('displayFooter')` → `registerHook('displayModalContent')`. **Since the module was already installed, `install()` changing doesn't retroactively fix the DB** — the `hook_module` row was migrated manually: `DELETE FROM hook_module WHERE id_module=87 AND id_hook=50 /* displayFooter */; INSERT INTO hook_module (id_module, id_shop, id_hook, position) VALUES (87, 1, 1122 /* displayModalContent */, 1);`. Anyone reinstalling this module from scratch gets it correctly via `install()` alone; upgrading an existing install of this module elsewhere would need the same manual fixup (or a proper upgrade script, not yet written — see ORPHANS).
5. **Auto-opening the modal does not use `bootstrap.Modal` / `window.bootstrap` at all — confirmed there is no such global anywhere on the site.** The custom theme (`themes/custom/`) only ever imports Bootstrap as an ES module inside its own Vite bundle (e.g. `src/js/quickview.ts` does `import {Modal} from 'bootstrap'`); it exposes no classic/UMD Bootstrap bundle and no `window.bootstrap`. Since `views/js/restore-cart-modal.js` is a plain classically-loaded script registered via `registerJavascript()` (outside the theme's build entirely), it cannot `import` anything.
   - The fix: **`restore-cart-modal.tpl` itself carries no `data-bs-toggle`/`data-bs-target` at all** — it's a plain, ordinary Bootstrap `.modal`, matching how the theme's own real modals (quickview, product-comments) are marked up (they never put those attributes on the modal element itself either; only on a separate trigger button elsewhere). The self-reference trick lives entirely in JS instead, and only for the instant it's needed: `modalElement.setAttribute('data-bs-toggle', 'modal')` + `setAttribute('data-bs-target', '#' + modalElement.id)`, then `dispatchEvent(new MouseEvent('click', {bubbles: true}))` — caught by Bootstrap's own document-level data-api listener (registered as a side effect of the Modal component being loaded anywhere on the page, e.g. via quickview), the same native mechanism already used for the `data-bs-dismiss="modal"` buttons — then both attributes are immediately `removeAttribute()`'d again, all synchronously (no other click can land in between `dispatchEvent()` and the removals, since `dispatchEvent()` itself is synchronous). No extra DOM element (hidden button or otherwise) was added, per explicit instruction — considered and rejected: (a) a hidden trigger button, (b) manually reimplementing Bootstrap's show/hide behavior in plain JS instead of using its real API.
   - **Why the attributes can't just stay set (one-shot gotcha, hit live and fixed):** while they're present on the modal element, *every* click inside the open modal (a checkbox, the add/dismiss buttons, anything) bubbles up through that same element, which still matches `[data-bs-toggle="modal"]` — Bootstrap's delegated listener treated each one as another toggle and closed the modal on the very first interaction. Removing them right after the initial dispatched click fixes this. The `data-bs-dismiss="modal"` buttons are unaffected since they carry their own attribute, independent of the modal element's.
5. Once the customer acts, both actions end by **deleting the abandoned cart** so it is never proposed again. Since there's no cookie anymore, `restore.php` gets `id_abandoned_cart` from the client: `restore-cart-modal.tpl` renders it into `data-id-abandoned-cart` on the modal element (from `mergesavedcart_abandoned_cart_id`, set alongside `mergesavedcart_products`/`mergesavedcart_restore_url` in `hookDisplayModalContent()`), `views/js/restore-cart-modal.js` reads that attribute once at load and includes it as `id_abandoned_cart` in every `postAction()` call, and `postProcess()` reads it via `Tools::getValue('id_abandoned_cart')` and threads it through `processAdd()`/`processDismiss()`/`deleteAbandonedCart()` as a plain parameter:
   - **Add to cart** → POSTs to `controllers/front/restore.php` (`action=add`, `products`, `id_abandoned_cart`), which calls `Cart::updateQty()` for each *checked* line on the customer's current cart, logs the count via `PrestaShopLogger::addLog()`, then calls `AbandonedCartFinder::deleteAbandonedCart()`, then reloads the page. **The "select all" toggle checkbox was removed, per explicit instruction — but the per-product checkboxes stay**, checked by default (`checked` attribute in the template), letting the customer deselect individual items before adding; `views/js/restore-cart-modal.js` filters `.mergesavedcart-product-checkbox` elements by `.checked` when building the `products` payload (a brief earlier revision removed the per-product checkboxes too and always added everything unconditionally — that was a misreading of the instruction, corrected back to per-product opt-out).
   - **No, thanks** → POSTs `action=dismiss` (`id_abandoned_cart`); logs the decision, no cart mutation, then also deletes the abandoned cart.
   - `AbandonedCartFinder::deleteAbandonedCart($idCustomer, $idAbandonedCart)` re-checks `Cart::id_customer` matches the logged-in customer before calling `Cart::delete()` (which itself refuses if an order now exists for that cart). With the cookie gone, this ownership check is now the *only* thing standing between a client-submitted id and an actual delete — there's no transport-level protection anymore, so this check must never be dropped or weakened.
6. **The modal cannot be dismissed except by clicking "No, thanks", per explicit instruction — forces an explicit choice.** `data-bs-backdrop="static"` (clicking outside no longer closes it, just flashes) and `data-bs-keyboard="false"` (Escape key no longer closes it) on the modal `<div>`; the header's `.btn-close` (X) button was removed entirely rather than left present-but-inert, since a visible close affordance that silently does nothing would be worse than not having one. "Add to cart" isn't wired to `data-bs-dismiss` either — it closes the modal only as a side effect of the subsequent `window.location.reload()`, not by dismissing it in place.
7. **`list-group-flush`** on the product `<ul>` — Bootstrap's built-in modifier that drops the list's own outer border/rounding and the left/right borders on each item, keeping only the horizontal divider between items, per explicit instruction ("only a separator between items").
8. A short explanatory `<p class="text-muted mb-3">` sits at the top of `.modal-body`, above the product list, telling the customer what the modal is and what the two footer buttons do — added per explicit instruction ("inform the user").
9. **Scoped scroll, not the whole-modal `modal-dialog-scrollable` pattern the theme's own quickview modal uses.** When there are more than 4 products (`{if $mergesavedcart_products|@count > 4}`), the `<ul class="list-group">` itself gets an inline `max-height: 430px; overflow-y: auto;` — only the product list scrolls, the informative paragraph above it and the footer buttons stay fixed/visible. `modal-dialog-scrollable` (Bootstrap's built-in modifier, confirmed already used by this theme's `quickview__dialog`) was considered and rejected here since it scrolls the *entire* `.modal-body`, including that paragraph, which is less precise than "scroll for the products" as instructed. `320px` is an approximation (~4 items at this template's current `py-3` + 64px-image row height), not derived from a measured value — revisit if the row markup changes size. This is the one inline style in the module; no custom CSS file was reintroduced (see the earlier z-index reversal note above) since Bootstrap has no built-in utility for "cap to N items then scroll."


**Why `config/admin/services.yml` + `config/front/services.yml` instead of one `config/services.yml`:** the first working version of this module put both services under a single module-root `config/services.yml`. This produced a live `ServiceNotFoundException: MergeSavedCart\Service\AbandonedCartFinder` when `hookActionAuthentication()` (fired via `Hook::exec` from `CustomerLoginForm::submit()`) tried `$this->get(AbandonedCartFinder::class)`. Clearing every relevant cache (`var/cache/dev/FrontContainer.php`, the `active_modules` cache pool, `AdminKernelDevDebugContainer.php`) did not fix it — confirming this wasn't a stale-cache problem. Comparing against `optimizer` (decorator, lives in `config/admin/services.yml`, short dotted service ids like `optimizer.translation.language_copier`) and `productcomments` (repository service, lives in `config/front/services.yml`, accessed via `$this->context->controller->getContainer()->get('product_comment_repository')` — a short snake_case id, not the FQCN) showed neither reference module relies on a bare module-root `config/services.yml` at all. Restructured to match both exactly: the BO-only decorator moved to `config/admin/services.yml`, the front-usable finder moved to `config/front/services.yml`, both now registered under short dotted ids (`mergesavedcart.customer_configuration_guard`, `mergesavedcart.abandoned_cart_finder`) instead of their FQCN, and front-hook access switched from `$this->get(AbandonedCartFinder::class)` to `$this->context->controller->getContainer()->get('mergesavedcart.abandoned_cart_finder')` (`restore.php`, already a `ModuleFrontController`, uses its own inherited `$this->get(...)` instead).

**Alignment with reference modules:**
- No `controllers/admin/`, no `Tab`, no `HelperForm`/`ObjectModel` — this feature only guards an *existing* core BO screen and adds FO behavior, so none of the legacy admin-form scaffolding used by `nw_multidomainmanager`/`nw_usp`/`wwwnutriconnect` was needed.
- The one piece of "modern Symfony" architecture (a decorated core service in `config/admin/services.yml`) directly mirrors `optimizer`'s `CustomLanguageFieldsFormDataHandler` decoration of `language_form_data_handler` — same shape, same file location, different target service.
- The front-usable `AbandonedCartFinder` service (`config/front/services.yml`, accessed via `$this->context->controller->getContainer()->get('mergesavedcart.abandoned_cart_finder')`) mirrors `productcomments`' `product_comment_repository` pattern exactly.
- Hook-driven front rendering (`hookDisplayModalContent`, `hookActionFrontControllerSetMedia`) mirrors the widget/asset-enqueue conventions in `nw_usp`/`wwwnutriconnect` — though `displayModalContent` itself is this theme's own hook point, not a pattern taken from either reference module.
- Smarty translations in `restore-cart-modal.tpl` use `{l s='...' d='Modules.Mergesavedcart.Shop'}` (explicit domain), not `{l s='...' mod='mergesavedcart'}` (the shorthand `nw_usp`/`wwwnutriconnect` use in their hook templates) — matching the convention in core-bundled `ps_wirepayment`'s hook templates (`d='Modules.Wirepayment.Shop'`), and consistent with the `Modules.Mergesavedcart.Admin` domain already used by `$this->trans()` calls in `mergesavedcart.php`.
- **Optional dependency on `nw_multidomainmanager`** (TLD scoping, see Rule 2/3 above): guarded with `Module::isEnabled('nw_multidomainmanager')` before touching `NwTld`, the same convention `wwwnutriconnect` uses (its `CLAUDE.md`: "always guard with `Module::isEnabled(...)`/`Module::isInstalled(...)`"). Not declared as a composer/module dependency — this codebase has no precedent for that; the guard is the only thing that makes it optional.

## [ACTION_STATUS]
Tracks the two customer-facing actions from the restoration modal. This section must be updated every time either action's behavior changes (per standing instruction) — do not let it drift from `controllers/front/restore.php`.

`id_abandoned_cart` is submitted by the client on both endpoints (`Tools::getValue('id_abandoned_cart')` in `postProcess()`) — no cookie involved anymore; see Rule 2/3.

| Action | Endpoint | Cart mutation | Abandoned cart deleted after | Logged via PrestaShopLogger | Status |
|---|---|---|---|---|---|
| Add to cart (per-product checkboxes, checked by default) | `restore.php` (`action=add`, `id_abandoned_cart`) | `Cart::updateQty()` per checked product on current cart | Yes — `AbandonedCartFinder::deleteAbandonedCart()`, ownership-checked | Yes — customer id + count added, context `Cart` | Implemented, not yet BO/manual-tested |
| Ignore / dismiss | `restore.php` (`action=dismiss`, `id_abandoned_cart`) | None | Yes — same deletion call | Yes — customer id, decision only | Implemented, not yet BO/manual-tested |

## [ORPHANS & PENDING]
- **Live bug report, fix applied but NOT yet confirmed: "Add to cart" reported `added: 3` (products count updated), but the 3 products were not actually found in the cart afterwards, and a stale "3" persisted in some on-page total even after logging out.** Investigated via direct DB queries, but the `cart`/`cart_product` tables had been reset to empty (`MAX(id_cart)` back down to 4) between the bug occurring and the investigation, while `qrrfl_log` still had the old `MergeSavedCart: customer #1 imported 3 product(s)...` entries (`id_log` 547, 566) — so the exact cart state at the time of the bug could not be forensically inspected; this fix is reasoned from precedent, not confirmed against the actual failure.
  - **Fix applied**: `views/js/restore-cart-modal.js`'s `fetch()` call was missing `credentials: 'same-origin'` and the `X-Requested-With`/`Accept` headers that `themes/custom/src/js/quickview.ts`'s own working `fetch()` call sends for its ajax add-to-cart-like flow. Without explicit `credentials`, an ajax request on a shop with multiple shop URLs (`mymuscle.prod:8888`, `.fr:8888`, `.it:8888` — confirmed in `qrrfl_shop_url`) risks not reliably carrying the session cookie, which would make `restore.php`'s `$this->context->cart` resolve to a different (or freshly-created) cart than the one the customer actually sees on the next page load — `Cart::updateQty()` would still report success (it succeeds on *whatever* cart `$this->context->cart` resolves to), matching the reported `added: 3` with nothing landing in the cart the customer checks afterward.
  - **Safety net also added**: `restore.php::deleteAbandonedCart()` now refuses to delete when `$idAbandonedCart === (int) $this->context->cart->id` — previously nothing stopped the abandoned cart from being deleted even if it happened to be the same cart `updateQty()` had just added products to, which would have silently erased them.
